# Generate Technical Documentation for Folder

Analyze code in `$folder` and generate/update technical documentation.

## Usage

```
/generate-docs src/features/auth
/generate-docs packages/shared/ui
```

**Argument** (`$folder`): Path to target folder (TARGET_DIR), relative to project root.

---

## What to Analyze

### Include
- Code files: `*.ts`, `*.tsx`, `*.js`, `*.jsx`, `*.py`, `*.go`, etc.
- Logic files: hooks, services, utils, models, controllers, reducers
- Tests: use for understanding behavior (scenarios, invariants), not detailed documentation
- README.md and Markdown files: for context, but **trust code over docs** if conflicts

### Ignore
- `node_modules`, `dist`, `build`, `out`, `.next`, `.turbo`, `.cache`, `coverage`
- Generated files: `*.generated.*`, `*.g.ts`, `*.g.dart`, `*.pb.go`
- Files with comments: `/* AUTO-GENERATED */`, `// generated by ...`
- Binary files: images, fonts, archives
- Config files (`package.json`, `tsconfig.json`, `vite.config.ts`, Dockerfile, CI configs): use only for
  understanding stack, don't document separately

---

## Output

Create/update: `TARGET_DIR/docs/index.md`

## Style & Structure

Follow [`../_shared/markdown-doc-principles.md`](../_shared/markdown-doc-principles.md)
for what to include/exclude, the size budget, the code-vs-doc authority note, and
the optional `docs/index.md` skeleton. Do not duplicate those rules here, and do
not reintroduce version tables, large code samples, or audit-report prose.

---

## Update Rules

If `docs/index.md` already exists:

1. **Don't rewrite from scratch** — analyze existing content first
2. **Identify sections** that are outdated and contradict current code, or have substantial gaps
3. **Update only what needs updating**: fix inaccuracies, add missing key aspects
4. **Preserve the author's custom sections** if they don't contradict code and carry useful context

---

## Analysis Depth

Read enough for effective work, not fanatically:

- Read entry points (`index.tsx`, `main.ts`, router, `server.ts`), main components/services, domain models,
  shared hooks/utils.
- Use tests to learn primary scenarios, edge cases, and protected invariants.

**Goal**: the next developer or agent reads `docs/index.md` first, then dives into code.

---

## Command Behavior

1. Analyze TARGET_DIR structure and content with the filters above
2. Determine module type (frontend/backend/library/mixed) and adapt emphasis
3. Create or update `TARGET_DIR/docs/index.md` per the shared principles
4. Keep the document lean, stable in structure, and focused on: what the module is, where the main logic
   lives, how to safely extend it, and which invariants must hold

All documentation must be in **English**.

---

## Error Handling

If TARGET_DIR doesn't exist or isn't accessible:
- Briefly inform the user
- Don't invent paths or analyze random folders
- Work automatically without clarifying questions in all normal cases
